# Windows Native 환경 (PowerShell 7) 기반 AI 에이전트 하네스 및 Gemini CLI 구축 아키텍처

## 1. 아키텍처 개요 및 권장 패턴 (2026년 기준)

WSL2(Windows Subsystem for Linux 2) 내부에 격리된 리눅스 환경을 구축한 것에 이어, Windows 호스트(Host) 운영체제에서 직접 구동되는 최신 **PowerShell 7 (pwsh)** 환경에 Gemini CLI를 구축하는 과정입니다. 

기존의 내장형 Windows PowerShell 5.1(파란색 화면)은 레거시(Legacy) 기술로 간주되어 최신 Node.js 패키지나 비동기 AI 에이전트 스크립트 실행 시 호환성 오류를 발생시킵니다. 따라서 크로스 플랫폼(Cross-platform)을 지원하는 **PowerShell 7 (검은색 화면)**을 기반으로 **NVM for Windows (Windows용 노드 버전 관리자)**를 활용하여 네이티브 환경을 세팅하는 것이 공식 보안 및 시스템 아키텍처 권장(Recommend) 패턴입니다.

* **Host (호스트)**: 가상 머신이나 서브시스템(WSL2)을 실행하는 바탕이 되는 물리적 또는 주 운영체제 환경(여기서는 Windows)을 의미합니다.
* **Legacy (레거시)**: 과거에 개발되어 현재의 새로운 기술 표준이나 보안 요구사항을 충족하지 못하지만, 호환성을 위해 남겨둔 낡은 시스템이나 소프트웨어를 뜻합니다.
* **Cross-platform (크로스 플랫폼)**: Windows, Linux, macOS 등 다양한 운영체제에서 공통적으로 실행될 수 있는 소프트웨어 환경입니다.

## 2. 단계별 구축 파이프라인 (Step-by-Step Pipeline)

모든 단계는 실행해야 할 정확한 **메뉴 위치**, **실행 권한(관리자 여부)**, 그리고 **프롬프트 경로**를 엄격하게 구분하여 명시합니다.

### 2.1. PowerShell 7 관리자 권한 실행 및 스크립트 실행 정책 완화

최초 환경 설정 시에는 Windows의 기본 보안 정책이 외부 스크립트 실행을 차단(Restricted)하고 있으므로, 이를 해제하기 위해 반드시 관리자 권한이 필요합니다.

* **실행 권한**: **관리자 모드 (Administrator Mode)** - 필수
* **메뉴 위치 및 정확한 실행 방법**:
    1.  Windows 바탕화면 하단 작업 표시줄의 **[시작(Start)]** 버튼을 클릭하거나 키보드의 `Win` 키를 누릅니다.
    2.  검색창에 `pwsh` 또는 `PowerShell 7`을 입력합니다.
    3.  검색 결과에 나타난 검은색 아이콘의 **[PowerShell 7 (x64)]** 항목 위에 마우스 커서를 올리고 **오른쪽 버튼**을 클릭합니다.
    4.  나타나는 컨텍스트 메뉴에서 **[관리자 권한으로 실행(Run as administrator)]**을 클릭합니다.
    5.  UAC (User Account Control, 사용자 계정 컨트롤) 경고창이 나타나면 **[예(Yes)]**를 클릭하여 진입합니다.
* **현재 프롬프트 위치**: `C:\Windows\System32>` (관리자 권한 실행 시 기본 진입 경로)

```powershell
# 현재 Windows 시스템의 스크립트 실행 정책(Execution Policy)이 무엇으로 설정되어 있는지 상태를 확인합니다.
Get-ExecutionPolicy
# 시스템 보안을 유지하면서도 로컬에서 작성하거나 신뢰할 수 있는 외부 스크립트(Node.js 설치 도구 등)를 실행할 수 있도록 정책을 'RemoteSigned'로 변경합니다.
Set-ExecutionPolicy RemoteSigned -Scope CurrentUser -Force
# 변경된 실행 정책이 현재 사용자 환경(CurrentUser)에 정상적으로 적용되었는지 다시 한번 검증합니다.
Get-ExecutionPolicy -List
```

* **UAC (User Account Control, 사용자 계정 컨트롤)**: 악성 코드나 비인가 프로그램이 시스템의 핵심 설정을 임의로 변경하지 못하도록, 관리자 권한이 필요한 작업 시 사용자에게 명시적으로 동의를 구하는 Windows 보안 기능입니다.
* **RemoteSigned (리모트 사인드, 원격 서명)**: 로컬 컴퓨터에서 작성된 스크립트는 제한 없이 실행하지만, 인터넷에서 다운로드한 스크립트는 신뢰할 수 있는 배포자의 디지털 서명이 있어야만 실행을 허용하는 중간 단계의 보안 정책입니다.

### 2.2. NVM for Windows (Windows용 노드 버전 관리자) 설치

Windows 네이티브 환경에서는 WSL2에서 사용했던 셸 스크립트 기반의 NVM이 작동하지 않으므로, Windows 전용으로 포팅된 `nvm-windows`를 `winget` 패키지 관리자로 설치합니다.

* **실행 권한**: **일반 사용자 모드** 또는 **관리자 모드** (현재 띄워둔 관리자 창을 그대로 사용해도 무방함)
* **현재 프롬프트 위치**: `C:\Windows\System32>` 또는 `C:\Users\사용자명>`

```powershell
# Windows의 공식 커맨드라인 패키지 관리자인 winget을 사용하여 'NVM for Windows' 패키지를 검색하고 조용히(자동으로) 설치합니다.
winget install -e --id CoreyButler.NVMforWindows --accept-package-agreements --accept-source-agreements
# NVM 설치 시 환경 변수(Environment Variables)가 레지스트리에 등록되지만 현재 열려있는 PowerShell 세션에는 즉각 반영되지 않으므로, 시스템 환경 변수를 강제로 다시 읽어와 현재 셸에 적용(Reload)합니다.
$env:Path = [System.Environment]::GetEnvironmentVariable("Path","Machine") + ";" + [System.Environment]::GetEnvironmentVariable("Path","User")
# nvm 명령어가 시스템 환경 변수에 정상적으로 매핑되었는지 버전 출력을 통해 검증합니다.
nvm version
```

### 2.3. Node.js 최신 LTS 버전 설치 및 심볼릭 링크(Symlink) 활성화

NVM for Windows를 통해 Node.js를 설치하고 활성화할 때, Windows 아키텍처 특성상 내부적으로 심볼릭 링크(Symbolic Link)를 생성해야 하므로 **반드시 관리자 권한**이 요구됩니다.

* **실행 권한**: **관리자 모드 (Administrator Mode)** - 필수 (2.1 단계에서 연 창을 계속 사용)
* **현재 프롬프트 위치**: 경로 무관

```powershell
# 현재 사용 가능한 Node.js의 모든 버전 목록 중 안정화된 LTS(Long Term Support) 버전 목록을 화면에 출력하여 확인합니다.
nvm list available
# 최신 안정화 버전인 Node.js LTS 버전(2026년 기준 최신)을 인터넷에서 다운로드하여 설치합니다.
nvm install lts
# 시스템에 여러 버전이 있을 경우 방금 설치한 'lts' 버전을 현재 시스템의 활성화된 기본 Node.js 런타임으로 지정(심볼릭 링크 연결)합니다.
nvm use lts
# 시스템에 설치된 전체 Node.js 버전 목록을 출력하고, 현재 활성화된 버전에 '*' 표시가 있는지 확인합니다.
nvm list
# Node.js 런타임이 정상적으로 실행되는지 버전을 출력하여 최종 검증합니다.
node -v
# Node.js와 함께 설치되는 패키지 관리자인 NPM(Node Package Manager)의 버전을 확인합니다.
npm -v
```

* **Symbolic Link (심볼릭 링크, 기호화된 링크)**: 운영체제의 파일 시스템에서 특정 파일이나 디렉토리를 가리키는 포인터(참조) 역할을 하는 특수한 파일로, Windows의 '바로 가기'와 유사하지만 시스템 커널 단에서 원본 파일과 동일하게 취급되는 기술입니다.

### 2.4. Gemini CLI 전역(Global) 설치 및 하네스 환경 세팅

Node.js 환경이 완벽히 구성되었으므로, NPM을 이용해 Gemini CLI를 Windows 시스템 전역에 설치합니다.

* **실행 권한**: 일반 사용자 모드 또는 관리자 모드 모두 가능
* **현재 프롬프트 위치**: 경로 무관

```powershell
# NPM을 사용하여 Google의 공식 Gemini CLI 패키지를 Windows 시스템 전역(-g 옵션)에 설치하여 모든 경로에서 명령어를 사용할 수 있게 합니다.
npm install -g @google/gemini-cli
# 설치된 패키지의 모듈 트리를 갱신하고 의존성 캐시를 정리하기 위해 환경을 새로고침합니다. (PowerShell 특성상 전역 패키지 설치 후 경로 인식을 위해 권장됨)
$env:Path = [System.Environment]::GetEnvironmentVariable("Path","Machine") + ";" + [System.Environment]::GetEnvironmentVariable("Path","User")
# PowerShell의 Get-Command (gcm) cmdlet을 사용하여 gemini 명령어의 실제 실행 파일 위치(경로)를 추적하고 유효성을 검증합니다.
Get-Command gemini
# Gemini CLI의 내장 버전 확인 명령을 호출하여 백엔드 AI 에이전트와 통신할 준비가 되었는지 테스트합니다.
gemini --version
```

## 3. 실전 팁 (Real-world Tips): 아키텍트 관점에서의 Windows PowerShell 7 최적화

* **WSL2와의 연동(Interop) 패턴 (매우 중요)**: 
    Windows 네이티브(PowerShell)에 설치된 Gemini CLI와 리눅스(WSL2) 내부에 설치된 Gemini CLI는 **물리적으로 완전히 분리된 별개의 환경**입니다. 만약 PowerShell 7 창에서 리눅스 안에 만들어둔 하네스나 셸 스크립트를 조작하고 싶다면, PowerShell 프롬프트에서 명령어 앞에 `wsl`을 붙여서 실행하는 **하이브리드 파이프라인(Hybrid Pipeline)** 방식을 사용해야 합니다.
    ```powershell
    # PowerShell에서 WSL2 내부의 리눅스용 gemini cli를 직접 호출하여 리눅스 환경의 파일을 분석하는 하이브리드 명령어 패턴입니다.
    wsl gemini run --prompt "리눅스의 /var/log/syslog 분석해줘"
    ```
* **PowerShell 프로파일(Profile)을 이용한 Alias(별칭) 설정**:
    모의해킹 실무에서는 명령어를 타이핑하는 1초의 시간도 아껴야 합니다. PowerShell 7의 사용자 프로파일(`$PROFILE`)을 열고 `Set-Alias gem -Value gemini` 형태의 코드를 주입해두면, 긴 명령어 대신 `gem` 단 두세 글자만으로 AI 에이전트를 즉각 호출할 수 있는 강력한 무기가 됩니다.

모의해킹 및 대규모 보안 에이전트 하네스(Harness)를 구축할 때, 이 두 환경은 각각 **'실행 샌드박스(Execution Sandbox)'**와 **'중앙 통제실(Control Plane)'**의 역할을 나누어 수행하게 됩니다. 그 명확한 이유와 아키텍처적 목적을 세 가지로 정리해 드립니다.

### **1. 운영체제(OS) 종속적인 보안 도구의 완벽한 분리**

가장 큰 이유는 모의해킹에 사용되는 대부분의 강력한 오픈소스 보안 도구들이 철저하게 **리눅스(Linux) 생태계**에 맞춰져 있기 때문입니다.

* **WSL2 (Linux) 환경의 역할 - '전술 요원(Worker Agent)'**:
    Nmap의 고급 스캐닝, Metasploit(메타스플로잇), OSS-Fuzz, 각종 커스텀 파이썬 익스플로잇(Exploit) 스크립트들은 Windows 네이티브 환경에서는 컴파일이 안 되거나 네트워크 패킷 제어 권한 문제로 정상 작동하지 않습니다. 따라서 WSL2 내부의 Gemini CLI는 철저히 **"리눅스 전용 보안 툴을 직접 실행하고 로그를 분석하는 현장 요원"**으로 세팅되어야 합니다.
* **Windows (PowerShell 7) 환경의 역할 - '지휘관(Supervisor Agent)'**:
    반면 Windows 환경의 Gemini CLI는 리눅스용 해킹 툴을 직접 돌리지 않습니다. 대신 호스트 PC의 전체 리소스, Active Directory(액티브 디렉토리) 자격 증명, 사내망 라우팅 테이블 등을 분석하거나, 전체 다중 에이전트 시스템(MAS)을 총괄하는 **"중앙 오케스트레이터(Orchestrator)"** 역할을 수행합니다.

### **2. 보안 샌드박스(Sandbox)를 통한 호스트 격리 및 안전 확보**

보안 취약점을 분석하고 악성코드나 익스플로잇 코드를 AI 에이전트가 자동 생성하여 테스트하는 '퍼징 하네스(Fuzzing Harness)'를 구동할 때, 이를 Windows 메인 운영체제에서 직접 실행하는 것은 매우 위험합니다.

* **격리된 환경(Isolation)**: WSL2는 가상화된 리눅스 커널 위에서 돌아가므로, AI 에이전트가 생성한 공격 코드가 폭주하거나 치명적인 시스템 파일 삭제 실수를 하더라도 Windows 호스트 OS에는 전혀 영향을 미치지 않습니다.
* 즉, WSL2는 언제든 초기화하고 부술 수 있는 **안전한 모의해킹 전용 실습장(Target & Sandbox)**이고, Windows PowerShell의 Gemini CLI는 그 밖에서 샌드박스를 안전하게 관찰하고 통제하는 역할을 합니다.

### **3. 하이브리드 파이프라인 (Hybrid Pipeline) 통신**

두 환경이 각자 구축되어 있으면, Windows의 PowerShell 7에서 다음과 같은 크로스 플랫폼(Cross-platform) 명령을 통해 리눅스 환경의 에이전트를 원격 조종하듯 다룰 수 있습니다.

```powershell
# Windows PowerShell에서 실행하지만, 실제 작업은 WSL2 내부의 리눅스용 에이전트가 격리된 상태로 수행합니다.
wsl gemini run --profile linux-recon-agent --prompt "리눅스 샌드박스 내부의 /var/log/auth.log 를 분석해서 무차별 대입 공격(Brute-force) 흔적을 찾아내."
```

* **Cross-platform (크로스 플랫폼)**: Windows와 Linux라는 서로 다른 운영체제 환경의 경계를 허물고 명령어와 데이터를 자유롭게 주고받는 기술적 패턴입니다.
* **Brute-force (무차별 대입 공격, 브루트 포스)**: 성공할 때까지 가능한 모든 비밀번호 조합을 반복적으로 시스템에 입력하여 로그인 암호를 크랙하는 해킹 기법입니다.

### **요약: 왜 두 번 설치해야 했는가?**

1.  **WSL2 내부의 Node.js/Gemini CLI**: 리눅스 전용 모의해킹 툴과 연동하여 격리된 환경에서 위험한 작업을 수행하는 **작업자(Worker)** 용도입니다.
2.  **Windows의 NVM/Gemini CLI**: 시스템 전체를 통제하고 WSL2 작업자들에게 명령을 하달하는 **관리자(Supervisor)** 용도입니다.

이러한 투 트랙(Two-Track) 구조가 2026년 현재 대규모 보안 아키텍처에서 AI 에이전트를 가장 안전하고 효율적으로 구동하는 표준 설계 방식입니다.

---3.1. Windows 수퍼바이저(Supervisor) 에이전트 적용 (PowerShell 7)
Windows 운영체제 시스템 환경 변수에 API Key를 영구적으로 등록하여, 향후 어떤 폴더에서 터미널을 열어도 즉시 통신이 가능하도록 세팅합니다.

실행 권한: 일반 사용자 모드 (사용자 환경 변수에 등록하므로 관리자 권한 불필요)

메뉴 위치 및 실행 방법: 바탕화면 검색창 -> pwsh 검색 -> [PowerShell 7] 일반 실행

현재 프롬프트 위치: 경로 무관

PowerShell
# [Environment]::SetEnvironmentVariable 메서드를 사용하여 'User' 계층의 환경 변수에 복사한 API Key를 영구 등록합니다.
# (주의: "발급받은_API_KEY_입력" 부분을 실제 복사한 AIzaSy... 문자열로 교체하여 실행하십시오.)
[Environment]::SetEnvironmentVariable("GEMINI_API_KEY", "발급받은_API_KEY_입력", "User")

# 환경 변수 등록 후, 현재 열려있는 셸(Shell) 세션에 즉시 적용하기 위해 변수를 리로드(Reload)합니다.
$env:GEMINI_API_KEY = [System.Environment]::GetEnvironmentVariable("GEMINI_API_KEY","User")

# Gemini 에이전트에 첫 번째 프롬프트를 전송하여 인증 및 통신 상태를 테스트합니다.
gemini run --prompt "Windows 환경 변수 인증 테스트. 현재 수퍼바이저 에이전트 통신 준비 완료를 보고해"
